## TL;DR

Sonnet 5 is a real jump — it beats Sonnet 4.6 on every published benchmark and now supports the same `xhigh` effort tier and Dynamic Workflows/`ultracode` orchestration that were Opus-exclusive until yesterday.The available effort levels for Sonnet 5, Opus 4.8, and Opus 4.7 are low, medium, high, xhigh, and max, while Opus 4.6 and Sonnet 4.6 top out at max without xhigh. It's now close enough to Opus 4.8 that Anthropic itself frames the choice as adjusting effort level between the two models to find the right cost/performance balance rather than picking one model for everything. Opus 4.8 still wins outright on the hardest, highest-stakes, longest-horizon work.

## Where Opus 4.8 is still clearly better

- **Accuracy ceiling on the hardest tasks.** Anthropic states plainly that Opus 4.8 is still the model of choice for higher accuracy on these tasks, and it leads Sonnet 5 by 4–6 points across SWE-bench Pro/Verified.
- **Judgment/self-correction on ambiguous, multi-service problems.** Opus 4.8 asks the right questions, catches its own mistakes, pushes back when a plan isn't sound, and builds confidence around complex multi-service explorations before making big changes.
- **Honesty/flagging uncertainty.** Opus 4.8 is four times less likely than Opus 4.7 to fail to report flawed code — a meaningful trait for infra/security-adjacent work where silent failure is expensive.
- **Cybersecurity-adjacent tasks.** Sonnet 5 was not deliberately trained on cybersecurity tasks and shows substantially poorer performance than Opus 4.8 on dangerous-cyber evaluations; Anthropic recommends Opus 4.8 for cybersecurity work requiring reduced guardrails. If you ever touch pentesting/exploit-adjacent work on your AWS infra, use Opus.
- **Massive, long-horizon autonomous migrations.** The Bun Zig→Rust port (~750K lines, 11 days, hundreds of parallel agents) ran on Opus 4.8's Dynamic Workflows. Sonnet 5 can now run the same feature, but Opus is the more battle-tested/reliable engine for that scale today.
- **At `xhigh`, cost can flip.** MarkTechPost's independent comparison flags that at xhigh effort, Sonnet 5's cost can exceed Opus 4.8 at similar quality — the token savings only materialize at low/medium/high, not maxed-out effort.

## Is Sonnet 5 good enough for coding now? Yes, with one real caveat

Yes for the large majority of day-to-day engineering work — brownfield debugging, multi-file refactors, test-writing, PR-sized changes, infra scripting. One founding engineer testimonial is telling: "Claude Sonnet 5 is at its best on brownfield code — race conditions, hidden tests, the parts nobody wants to touch. It traces a failure to its actual root cause and ships a durable fix instead of patching the symptom."

The caveat, straight from live Hacker News threads on launch day: some developers doing **interactive, human-in-the-loop coding** (not fully autonomous agent runs) are wary of heavier agentic optimization. One commenter noted they'd been using Sonnet 4.6 more than Opus specifically because they do agent-assisted rather than fully agent-driven development, and that models optimized for full autonomy tend to get worse at assisted development, doing too much despite strict instructions. If most of your Claude Code usage is tight, supervised, back-and-forth work (which a lot of your skill-ops/skillctl and Terraform work sounds like), test that Sonnet 5's new "finish the whole thing without asking" behavior doesn't overshoot your intent — Anthropic's own docs note Sonnet 5 provides regular higher-quality progress updates and will reach for tools/self-verification loops more readily than 4.6, and recommends removing scaffolding you'd previously added to force interim status updates.

## A few operational notes worth flagging

- **Tokenizer change**: Sonnet 5 uses a new tokenizer that produces roughly 30% more tokens for the same input text than Sonnet 4.6 — recount any hardcoded `max_tokens` budgets you migrate over, or you'll see truncated responses.
- **Pricing window**: $2/$10 per MTok introductory pricing runs through August 31, 2026, then standard pricing is $3/$15 — worth locking in heavy testing/migration now while it's cheap.
- **API breaking changes**: manual extended thinking (budget_tokens) and non-default temperature/top_p/top_k now return 400 errors on Sonnet 5 — if any of your Gmail API automation or Turo sync scripts set these explicitly, they'll break on migration.
