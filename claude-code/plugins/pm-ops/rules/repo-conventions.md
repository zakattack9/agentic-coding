# Central PM repo conventions

The contract every pm-ops artifact obeys. `lib/pm.py` enforces the mechanical
parts; this file is the human reference. (The scaffold's own copy lives at
`templates/repo-scaffold/CLAUDE.md` and ships into each PM repo.)

## Source of truth
Markdown-in-git is canonical. The board (GitHub Issues + Projects via the default
engine) is a **projection** produced by `route-task`. If the two disagree, the
markdown front-matter wins; re-route to reconcile.

## Folders (lifecycle stages)
`pm.py` knows these exact paths (`STAGE_DIRS`); the folder an item lives in *is* its stage:

| Stage | Path | Meaning |
|-------|------|---------|
| inbox | `inbox/` | Raw, unstructured dumps, verbatim. Not yet items. |
| backlog | `backlog/` | Consolidated **candidates** — one file each (`consolidate-backlog`). |
| draft | `specs/draft/` | Specs being authored / refined. |
| active | `specs/active/` | Refined, ready, or in-flight specs. |
| promoted | `specs/promoted/` | Forwarding **stubs** left after a spec moved into its impl repo. |

`stories/` does not exist on purpose — **everything is a spec** (one format,
scaled rigor; see `rigor-rubric.md`).

## Stable IDs
- Every item carries `id: <PREFIX>-NNNN` (default `PM-0042`), allocated by
  `pm.py new-id` from `.pm-ops/registry.json`.
- The id is **immutable** and **never changes as the file moves between folders**.
  It is the thread that ties backlog → spec → issue → PR.
- Filenames are `<id>-<kebab-slug>.md` (e.g. `PM-0042-bulk-discounts.md`). Renaming
  the slug is fine; the id prefix must stay.

## Front-matter contract
YAML front-matter is the engine-neutral "normalized task". Write **flow-style**
collections (`[a, b]`, `{k: v}`) so `pm.py` parses them without a YAML dependency.

| Key | Required | Values / notes |
|-----|----------|----------------|
| `id` | yes | `PM-NNNN`, immutable. |
| `title` | yes | Short imperative phrase. |
| `type` | yes | `feature` · `bug` · `chore` · `epic`. |
| `status` | yes | `inbox` · `candidate` · `draft` · `active` · `promoted` · `done`. |
| `size` | rec. | `S` · `M` · `L` · `XL`. |
| `priority` | rec. | `P0` · `P1` · `P2` · `P3`. |
| `rigor` | yes | `light` · `full` (see `rigor-rubric.md`). |
| `parent` | if child | Parent epic's id (`PM-0007`). |
| `depends_on` | opt. | List of ids this is blocked by (`[PM-0039]`). |
| `spec` | when placed | Repo-relative path to the spec file. |
| `impl_repo` | when known | `owner/name` of the target implementation repo. |
| `labels` | opt. | Free-form list; engines may map them (e.g. repo routing). |
| `assignees` | opt. | List of tracker usernames. |
| `board` | engine-written | `{engine, ref, url}` — written back by `route-task` after `upsert`. |
| `source` | backlog only | Where a candidate came from (`inbox/<file>`). |
| `created` | rec. | `YYYY-MM-DD`. |

Only the keys in `pm.py`'s `NORMALIZED_KEYS` reach the engine; others are
pm-internal and ignored by boards.

## Index
`index.md` is generated by `pm.py reindex` — a sorted table of every artifact with
its stage, status, rigor, board link, and file. **Never hand-edit it.** Re-run
reindex after any change that should be visible at a glance.

## `.pm-ops/`
- `config.json` — `engine` name + engine config block (for github: `owner`,
  `project`, `default_repo`, `repo_map`, field name mappings). Choosing the board
  is one edit here; see `engines/INTERFACE.md`.
- `registry.json` — `{prefix, next}` id counter. Managed by `pm.py new-id`.
